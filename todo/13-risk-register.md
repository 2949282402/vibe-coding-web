# 第一阶段风险台账与上线注意事项

## 一、目标

识别第一阶段实施过程中的主要风险，并给出规避、回滚和上线前检查建议。

## 二、核心风险清单

## 风险 1：`/knowledge` 页面重构影响现有问答体验

### 表现

- 页面结构虽然更清晰，但原有发送、流式渲染、历史记录逻辑被误伤。
- 首屏重构后用户无法快速继续历史会话。

### 规避方案

- 将“页面布局改造”和“SSE/会话逻辑改造”分步提交。
- 先做 UI 容器重构，再接入元信息字段。
- 保留历史会话入口，避免只剩营销式首页态。

### 回滚方案

- 若新首页态影响主流程，可仅保留 Agent 移除和 Ask 降级，暂时回退首页空态改造。

## 风险 2：新增响应字段导致前后端联调异常

### 表现

- 前端对新增字段依赖过强，后端部分场景未返回导致页面报错。
- SSE `done` 与普通接口返回结构不一致。

### 规避方案

- 新字段全部做可空兼容。
- 普通 `ask` 与 SSE `done` 统一走同一份响应组装逻辑。
- 联调前先完成接口样例确认。

### 回滚方案

- 保留旧字段消费路径，必要时前端隐藏元信息展示区但不回退接口升级。

## 风险 3：Agent 草稿流改造与当前发布逻辑冲突

### 表现

- Agent 完成后无法正常生成内容。
- 发布逻辑依赖旧状态，导致草稿无法审核发布。

### 规避方案

- 不重写现有发布服务，只在其前面增加人工审核层。
- 将“执行状态”和“审核状态”分离，避免一个字段承载全部语义。
- 先跑通 `DRAFT_READY -> PUBLISHED`，再补驳回分支。

### 回滚方案

- 若审核流异常，可保留草稿存储能力，临时关闭审核页入口，先不开放发布动作。

## 风险 4：数据库改造影响旧环境兼容性

### 表现

- 旧库缺字段、缺索引，部署后接口报错。
- 初始化脚本只适合新环境，不适合现有环境升级。

### 规避方案

- 在 schema initializer 中增加“存在性判断后再创建/补字段”。
- 所有新字段尽量允许空值或提供默认值。

### 回滚方案

- 先停用统计写入或审核字段使用，保留核心问答与发布主链路。

## 风险 5：SSE 在 nginx 或容器环境中退化

### 表现

- 本地开发正常，docker/nginx 环境下流式输出延迟或一次性返回。

### 规避方案

- 明确检查 `frontend/nginx/default.conf.template`。
- 每次涉及代理相关改动都做容器环境复测。

### 回滚方案

- 保持普通非流式问答可用，SSE 异常时前端可临时降级到普通接口。

## 风险 6：缓存失效不完整导致发布后前台不一致

### 表现

- 草稿已发布，但首页或文章列表仍看不到。
- 旧缓存导致用户看到过期内容。

### 规避方案

- 明确发布成功后的缓存失效点：首页、文章列表、详情。
- 驳回草稿时不做多余缓存失效。

### 回滚方案

- 上线初期如出现缓存不一致，可临时缩短 TTL 或手动清理相关缓存。

## 风险 7：权限边界收紧不彻底

### 表现

- 前台虽然隐藏了 Agent，但后端接口仍可能被普通用户访问。
- 新增草稿审核接口只做了前端限制，没有做后端权限限制。

### 规避方案

- 所有 `/api/admin/**`、`/api/agent/**` 审核动作继续走管理员权限校验。
- 前端路由守卫只作为辅助，后端权限才是最终边界。

### 回滚方案

- 发现权限问题时，优先关闭前端入口并在后端直接禁用相关接口。

## 三、上线前检查清单

- `/knowledge` 页面主流程可用
- Ask 与 RAG 模式切换正常
- SSE 在 docker/nginx 环境中正常流式输出
- Agent 生成后进入草稿，不直接发布
- 审核通过后可在前台看到新文章
- 非管理员无法访问审核接口
- 新表、新字段、索引已成功落库
- 日志中无明显新增异常

## 四、上线策略建议

### 1. 分步上线

建议顺序：

1. 先上线 RAG 响应字段扩展，前端先兼容不强依赖
2. 再上线 `/knowledge` 页面结构调整
3. 再上线 Agent 草稿审核流
4. 最后上线统计看板

### 2. 小步发布

- 每个子能力独立发布，不要把所有改动压成单次大上线。
- 先确保核心用户链路稳定，再补运营能力。

## 五、回滚原则

- 页面问题优先回滚前端展示，不先回滚接口结构。
- Agent 审核流异常时，优先关闭发布动作，不要直接恢复自动发布。
- 统计与埋点问题不应阻断问答主流程。

## 六、最终判断

第一阶段的风险不是“做不出来”，而是“改动点跨前端、后端、数据库、nginx，容易局部看起来对、整体联动出问题”。

因此执行上应坚持三条原则：

- 新字段向后兼容
- 状态机明确分层
- SSE、权限、缓存作为专项回归项单独验证
