# 产品优化建议总方案

## 一、产品结构调整

### 建议 1：重构 `/knowledge` 的产品定位

目标：让普通用户只面对清晰的问答场景。

建议改为：

- 前台 `/knowledge` 默认模式：`基于本站内容提问`
- 高级选项中再暴露搜索范围：`仅站内` / `站内+网络` / `仅网络`
- `Ask` 不再与 `RAG` 并列展示，改为“无知识库回答”高级选项
- `Agent` 从前台移除，迁移到后台独立工作台

收益：

- 降低用户认知负担
- 强化本站知识问答定位
- 防止管理员工具污染前台体验

### 建议 2：新增“知识问答首页态”

当前 `/knowledge` 更像聊天页，不像产品页。建议增加首页态模块：

- 示例问题
- 热门问题
- 最近更新的知识主题
- 支持能力说明：基于博客内容回答、支持引用、支持继续追问

收益：

- 提高首次使用成功率
- 降低空白页带来的使用门槛

## 二、RAG 与 Ask 的功能优化

### 建议 3：强化 RAG 回答的可信度表达

每条回答增加：

- 回答来源类型：站内 / 网络 / 混合 / 无明确来源
- 命中来源数量
- 主要依据文章
- 内容更新时间
- 可信度标签：高 / 中 / 低

并在低置信度时显式提示：

- 本次回答缺少充分站内证据
- 本次回答主要来自网络搜索
- 本次回答为模型生成，建议人工核验

### 建议 4：增加“引用即跳转”体验

当前引用更多是列表信息，建议升级为：

- 点击引用，直接跳转到相关文章
- 支持定位到对应段落或片段
- 在回答中高亮引用编号和片段摘要

### 建议 5：保留 Ask，但降级其产品地位

`Ask` 仍有价值，因为它解决“知识库无覆盖时仍可回答”的问题，但不应作为主模式。

建议：

- 前台默认不突出展示 Ask
- 在无命中或用户主动切换时再引导使用 Ask
- Ask 结果默认标记为“非站内事实回答”

## 三、Agent 能力调整

### 建议 6：取消 Agent 默认直接发布

改为：

- Agent 完成任务后默认生成草稿
- 进入后台草稿箱
- 管理员审核后再发布

这是最重要的风险控制改动。

### 建议 7：为 Agent 增加审阅面板

建议增加三类审阅结果：

- 结构审阅：标题、段落、层次是否完整
- 引用审阅：是否存在无依据断言
- 发布审阅：是否包含计划文本、工具痕迹、占位内容

### 建议 8：增加内容生产模式切换

Agent 可支持：

- 选题调研
- 生成文章大纲
- 生成草稿
- 润色已有草稿
- 生成摘要/SEO 描述/封面文案

这比单一“写一篇并发布”更符合真实内容生产流程。

## 四、后台运营能力优化

### 建议 9：新增“知识库运营”页面

建议展示：

- 已索引文章数、chunk 数
- 最近索引更新时间
- 问答总量、帮助率、平均响应时长
- 无命中问题 Top N
- 低帮助率问题 Top N
- Web Search 占比
- 回答失败率、SSE 中断率

### 建议 10：新增“内容生产工作台”

将分散的 Agent Ops、Tool Calls、Feedback 视图整合成工作台，包括：

- 当前任务
- 草稿箱
- 待审核内容
- 已发布内容回看
- 执行追踪与工具调用记录

## 五、旧功能处理建议

### 建议保留但优化

- `RAG`：保留，作为核心能力
- `Ask`：保留，但降级入口和标识
- SSE：保留，继续作为核心交互体验
- 反馈功能：保留，但接入运营看板

### 建议移除或弱化

- 前台中的 `Agent` 模式入口
- Agent 默认自动发布路径
- 用户对复杂搜索模式的直接暴露，改为渐进展示

## 六、用户价值排序

优先级从高到低：

1. 让普通用户更容易问到可信答案
2. 让管理员更稳定地生产内容
3. 让后台能看到问答和内容生产质量
4. 再考虑更复杂的新能力，例如推荐、个性化、知识卡片沉淀
