# 当前项目现状审视

## 一、现有能力概览

当前项目实际包含三条业务线：

- 内容站点：首页、文章、分类、归档、评论、简历静态页
- 知识问答：`/knowledge` 页面下的 `RAG` 与 `Ask`
- 内容生产：管理员 `Agent` 写作、执行追踪、工具调用、发布

技术上已经具备：

- Vue 3 + Vite 前端
- Spring Boot + MyBatis + Redis + MySQL 后端
- Qdrant 向量检索
- SSE 流式响应
- 基于用户配置的 Qwen 模型和 Web Search 能力

## 二、当前最核心的问题

### 1. 产品入口混杂

`/knowledge` 同时承载 `RAG`、`Ask`、`Agent` 三种模式。
这三者对应的用户目标完全不同：

- `RAG`：基于站内内容进行可信回答
- `Ask`：不依赖知识库的泛问答
- `Agent`：后台运营生产工具

将三者放在一个入口，会造成：

- 普通用户不知道该如何选择模式
- 管理员生产流程与前台体验耦合
- 产品心智不稳定，后续扩展困难

### 2. RAG 信任表达不足

当前系统虽已支持 `sources`、`citations`、`feedback`、`history`，但对用户来说仍然缺少“这条回答为什么可信”的清晰表达。
主要问题包括：

- 站内命中与网络搜索命中未被显式区分
- 无法快速判断回答是否主要来自本地知识
- 无法直观看到命中数量、内容新鲜度、证据强弱
- 低质量回答缺少明显兜底提示

### 3. Agent 发布风险过高

当前 Agent 具备写作和发布能力，虽然适合演示，但不适合长期稳定运营。
主要风险：

- 生成内容可能夹带计划文本、调试文本、无效引用
- 若默认直接发布，内容质量难以稳定控制
- 无草稿审阅和发布前检查，运营容错低

### 4. 缺少知识库运营闭环

现有系统已有反馈和历史记录，但还不够形成运营闭环。
目前缺少：

- 无命中问题统计
- 低帮助率问题分析
- Web Search 使用占比
- 问答延迟、失败率、SSE 中断率
- Agent 生成内容的采纳率和返工率

### 5. 后台能力碎片化

后台已经有 Agent Ops、Tool Calls、RAG Feedback 等页面，但这些能力更像运维视图，不像真正的“内容与知识运营工作台”。

## 三、产品判断

项目下一阶段不应继续以“再加一个模式”作为主方向，而应聚焦：

- 让普通用户更容易获得可信答案
- 让管理员更稳地生产和审核内容
- 让系统具备可观察、可迭代、可评估的运营能力

## 四、建议总纲

- 拆分前台问答与后台 Agent
- 弱化 `Ask` 的一级入口地位
- 增强 RAG 回答的可解释性
- 将 Agent 改为“默认生成草稿，人工确认发布”
- 增加知识库运营后台与指标埋点
