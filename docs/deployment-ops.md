# HeJulian Blog Deployment Guide

This document is for operations. The recommended deployment mode is `Docker Compose`, which starts `frontend`, `backend`, `mysql`, `redis`, and `qdrant` together.

The default RAG implementation now uses a standard pipeline:

```text
embedding recall -> rerank -> generation
```

The default model stack is:

- embedding: `text-embedding-v4`
- rerank: `qwen3-rerank`
- generation: `qwen-max`

## 1. Services

The project contains 5 core services and 1 optional bridge process:

- `frontend`: Vue 3 static assets served by Nginx
- `backend`: Spring Boot API service
- `mysql`: business database
- `redis`: cache service, also used by the RAG query cache
- `qdrant`: vector database for chunk embeddings and similarity retrieval
- `llm-bridge`: optional Python bridge service or host process for local Ollama experiments

Traffic path:

```text
Browser
  |
  v
frontend :80
  |-- /                -> Vue static assets
  |-- /api/**          -> backend:8080
  |-- /actuator/health -> backend:8080
  |
  v
backend :8080
  |-- MyBatis -> mysql:3306
  |-- Cache   -> redis:6379
  |-- Vector  -> qdrant:6333
  |-- RAG     -> text-embedding-v4 -> qwen3-rerank -> qwen-max
```

## 4. First-Time Startup

### 4.1 Prepare Environment File

```bash
cp .env.example .env
```

Recommended RAG variables:

```dotenv
BLOG_RAG_GENERATOR_ENABLED=true
BLOG_RAG_EMBEDDING_ENABLED=true
BLOG_RAG_RERANK_ENABLED=true
BLOG_RAG_QDRANT_ENABLED=true
BLOG_RAG_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
BLOG_RAG_MODEL=qwen-max
BLOG_RAG_GENERATOR_TIMEOUT_SECONDS=180
BLOG_RAG_EMBEDDING_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
BLOG_RAG_EMBEDDING_MODEL=text-embedding-v4
BLOG_RAG_EMBEDDING_DIMENSIONS=1024
BLOG_RAG_EMBEDDING_BATCH_SIZE=10
BLOG_RAG_RERANK_BASE_URL=https://dashscope.aliyuncs.com/compatible-api/v1
BLOG_RAG_RERANK_MODEL=qwen3-rerank
BLOG_RAG_QDRANT_BASE_URL=http://qdrant:6333
BLOG_RAG_QDRANT_COLLECTION=blog_knowledge_chunks
BLOG_RAG_RECALL_MULTIPLIER=4
BLOG_RAG_STREAM_TIMEOUT_MILLIS=900000
BLOG_RAG_MIN_SCORE=0.2
```

API key configuration:

- Recommended: set `DASHSCOPE_API_KEY` in the host environment before running `docker compose`
- Optional: fill `BLOG_RAG_API_KEY`, `BLOG_RAG_EMBEDDING_API_KEY`, and `BLOG_RAG_RERANK_API_KEY` in `.env`

### 4.2 Start

```bash
docker compose up -d --build
```

### 4.3 Verify

```bash
docker compose ps
curl http://127.0.0.1:${FRONTEND_PORT:-80}/actuator/health
curl http://127.0.0.1:${BACKEND_PORT:-8080}/actuator/health
```

Expected response:

```json
{"status":"UP"}
```

## 6. RAG Operations Notes

### 6.1 How RAG Works

- The backend stores all `PUBLISHED` post chunks in `rag_chunks`
- The backend writes dense vectors into `Qdrant`
- Query-time retrieval is executed by `Qdrant` nearest-neighbor search
- Candidate chunks are refined by `qwen3-rerank`
- Final answer is generated by `qwen-max`
- Redis caches repeated non-streaming RAG query responses
- Article create/update now triggers chunking and vector indexing immediately

### 6.2 Stream Mode

The public stream endpoint is:

```text
POST /api/public/rag/ask/stream
```

Streaming notes:

- Backend SSE timeout: `BLOG_RAG_STREAM_TIMEOUT_MILLIS`
- Generation timeout: `BLOG_RAG_GENERATOR_TIMEOUT_SECONDS`
- Frontend Nginx proxy buffering for this route is disabled

### 6.3 Optional: Local Ollama Through Python Bridge

If you want to bypass DashScope and use local Ollama, enable the optional bridge mode:

```dotenv
BLOG_RAG_GENERATOR_ENABLED=true
BLOG_RAG_EMBEDDING_ENABLED=false
BLOG_RAG_RERANK_ENABLED=false
BLOG_RAG_PYTHON_BRIDGE_ENABLED=true
BLOG_RAG_PYTHON_BRIDGE_BASE_URL=http://host.docker.internal:8090
BLOG_RAG_MODEL=qwen3.5:4b
OLLAMA_BASE_URL=http://host.docker.internal:11434
OLLAMA_TIMEOUT_SECONDS=600
```

Start the Python bridge on the host:

```bash
python llm-bridge/server.py
```

This mode is intended mainly for local experiments. The default production recommendation is DashScope.

## 8. Release and Rollback

Standard release:

1. Back up MySQL.
2. Pull the latest code.
3. Check whether `.env` changed.
4. Run `docker compose up -d --build`.
5. Verify homepage, login, `/knowledge`, and the RAG stream endpoint.

Rollback recommendation:

1. Tag every release.
2. Keep database backups.
3. Roll back code to the previous tag.
4. Run `docker compose up -d --build`.

## 9. Troubleshooting

### 9.1 Frontend Opens but API Returns 502/504

Check in order:

1. `docker compose ps`
2. `docker compose logs -f backend`
3. `curl http://127.0.0.1:8080/actuator/health`

### 9.2 Backend Fails to Start

Common causes:

- MySQL not ready
- wrong database credentials
- port conflict
- insufficient host memory
- missing RAG API key while external generation is enabled

### 9.3 RAG Is Only Returning Retrieval Answers

Check:

1. `BLOG_RAG_GENERATOR_ENABLED=true`
2. `BLOG_RAG_EMBEDDING_ENABLED=true`
3. `BLOG_RAG_RERANK_ENABLED=true` if rerank is expected
4. `DASHSCOPE_API_KEY` or the explicit API-key variables are available in the backend container
5. `docker compose logs -f backend` does not show upstream model API errors

### 9.4 MySQL Initialization Did Not Run

Confirm:

1. volumes were empty on first startup
2. [sql/blog_mysql_init.sql](/E:/coding/hejulian-web/sql/blog_mysql_init.sql) is mounted correctly
3. if re-initialization is needed in a non-production environment:

```bash
docker compose down -v
docker compose up -d --build
```
